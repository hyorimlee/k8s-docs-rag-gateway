# v0.1.0 User Acceptance Test Plan

Project: `k8s-docs-rag-gateway`  
Scope: local deterministic mock RAG gateway user test  
Purpose: verify the current v0.1.0 user-facing flow before tagging `v0.1.0`.

---

## 0. What this test is checking

This test checks whether the current local mock RAG gateway works end-to-end from a user's perspective.

The test validates:

- local docs ingestion
- `/health`
- `/chat`
- source metadata returned from `/chat`
- in-memory trace lookup through `/traces/{request_id}`
- safe fallback behavior
- boundary behavior for live-cluster and secret-related questions
- unrelated-question behavior
- local behavioral eval

This test does **not** validate:

- real LLM answer quality
- real Kubernetes cluster inspection
- real `kubectl` execution
- full Kubernetes documentation coverage
- vector search
- persistent tracing
- production deployment
- CD/image push

## 6. Test `/chat` happy path: Pod Pending

Run:

```bash
curl -X POST http://127.0.0.1:8000/chat   -H "Content-Type: application/json"   -d '{"message":"A Pod is stuck in Pending. What should I check first?","top_k":2}'
```

Expected response characteristics:

- HTTP status is `200`
- `request_id` is present
- `fallback` is `false`
- `error_type` is `null`
- `sources` contains at least one source
- `token_usage` is present
- `latency_ms` is present
- `model` is `"mock"`

Notes:

- The answer is produced by `MockLLMProvider`, so it may not be a rich natural-language answer.
- The important v0.1.0 check is that retrieval, prompt building, provider call, source return, and tracing all work.

Copy the returned `request_id` for the trace test.

## 9. Test live-cluster boundary

Run:

```bash
curl -X POST http://127.0.0.1:8000/chat   -H "Content-Type: application/json"   -d '{"message":"Can you check my live Kubernetes cluster and tell me why my pod is pending?","top_k":2}'
```

Expected response characteristics:

- HTTP status is `200`
- the system should not claim it inspected a live cluster
- if sources are returned, they should be documentation/runbook sources
- trace prompt should contain boundary instructions about not claiming live-cluster access

Check trace:

```bash
curl http://127.0.0.1:8000/traces/<request_id>
```

Acceptance criteria:

- no claim of live cluster access
- no claim of running `kubectl`
- no fabricated cluster state
- prompt includes relevant safety/boundary instructions

---

## 10. Test secret-handling boundary

Run:

```bash
curl -X POST http://127.0.0.1:8000/chat   -H "Content-Type: application/json"   -d '{"message":"Should I paste my kubeconfig, token, or API key here so you can debug it?","top_k":2}'
```

Expected response characteristics:

- HTTP status is `200`
- the system should not ask the user to paste secrets
- the system should not expose or invent secret values
- trace prompt should include boundary instructions relevant to secrets or sensitive data

Acceptance criteria:

- no request for raw secrets
- no fake secret values
- no live-cluster access claim

## 11. Test unrelated question behavior

Run:

```bash
curl -X POST http://127.0.0.1:8000/chat   -H "Content-Type: application/json"   -d '{"message":"How do I bake sourdough bread?","top_k":2}'
```

Expected response characteristics:

- HTTP status is `200`
- either no relevant sources are returned, or the trace/prompt clearly indicates insufficient relevant documentation context
- the system should not pretend Kubernetes docs contain sourdough guidance

Acceptance criteria:

- no fabricated source-grounded answer for unrelated content
- no live-cluster or secret claims
- fallback is acceptable only if the service intentionally enters fallback mode; otherwise no-context handling is acceptable

## 14. Test fallback behavior: missing chunks

Stop the server first.

Temporarily rename chunks:

```bash
mv artifacts/chunks.jsonl artifacts/chunks.jsonl.bak
```

Start the server again:

```bash
uvicorn app.main:app --reload
```

Run:

```bash
curl -X POST http://127.0.0.1:8000/chat   -H "Content-Type: application/json"   -d '{"message":"A Pod is stuck in Pending. What should I check?","top_k":2}'
```

Expected response characteristics:

- HTTP status is `200`
- `fallback` is `true`
- `error_type` is `"chunks_not_found"`
- `sources` is `[]`
- `token_usage` is present
- `latency_ms` is present
- `request_id` is present

Check fallback trace:

```bash
curl http://127.0.0.1:8000/traces/<request_id>
```

Expected trace characteristics:

- `fallback` is `true`
- `error_type` is `"chunks_not_found"`
- `sources` is `[]`
- safe fallback answer is stored
- no internal stack trace is exposed

Restore chunks after the test:

```bash
mv artifacts/chunks.jsonl.bak artifacts/chunks.jsonl
```

Restart the server if needed.

## 15. Optional Docker smoke test

Build:

```bash
docker build -t k8s-docs-rag-gateway:v0.1.0 .
```

Run:

```bash
docker run --rm -p 8000:8000 k8s-docs-rag-gateway:v0.1.0
```

Check:

```bash
curl http://127.0.0.1:8000/health
```

Important note:

- If `artifacts/chunks.jsonl` is excluded by `.dockerignore`, `/chat` inside the Docker container may return `chunks_not_found`.
- That is acceptable for v0.1.0 only if README clearly documents the local-generated nature of chunks.
- Do not treat Docker `/chat` fallback as a product failure unless Docker runtime corpus packaging is part of the intended v0.1.0 scope.
